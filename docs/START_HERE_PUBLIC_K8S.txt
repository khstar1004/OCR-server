Runtime layout:
- a-cong-ocr-app: web/demo UI, lightweight a-cong-ocr-ui image
- a-cong-ocr-playground: playground UI/proxy, lightweight a-cong-ocr-ui image
- a-cong-ocr-service: OCR API, heavy a-cong-ocr image
- a-cong-vllm-ocr: Chandra vLLM, GPU-only internal service
- Runtime settings file: /data/runtime/runtime-config/settings.json
- Account/session file: /data/runtime/runtime-config/auth.json
- Initial admin login: admin / roqkfrhk1! (override PLAYGROUND_ADMIN_PASSWORD in the Secret for production)

Change settings without image rebuild:
- Use the playground admin page, or
- Login through /a-cong-ocr-playground/api/auth/login and then GET/PUT /a-cong-ocr-playground/api/admin/runtime-settings with the session cookie.
- Live/new-request keys include OCR timeout, OCR concurrency, Chandra prompt/batch, playground page/upload limits, LLM post-processing URL, watcher polling, and target API URL/timeout.
- vLLM launch args such as VLLM_MODEL_NAME, VLLM_MAX_NUM_SEQS, VLLM_MAX_MODEL_LEN, VLLM_GPU_MEMORY_UTILIZATION, and VLLM_MM_PROCESSOR_KWARGS still require restarting only a-cong-vllm-ocr. The vLLM entrypoint reads the saved runtime JSON on startup.

Before applying:
1. Confirm image paths in the YAML match Harbor.
2. Set TARGET_API_BASE_URL in the ConfigMap.
3. Set TARGET_API_TOKEN in the Secret if the callback target needs it.
4. Confirm the model will exist at /models/chandra-ocr-2/config.json inside the pods.
5. Do not expose a-cong-vllm-ocr through Ingress.

New or rebuild deployment:
chmod +x scripts/*.sh
./scripts/deploy_public_ocr_closed_network.sh

This command runs preflight checks first:
- namespace nocodeaidev
- IngressClass nginx
- StorageClass local-path
- imagePullSecret harbor-reg-cred
- GPU node nocode-ai-army01
- free nvidia.com/gpu >= 1
- no Ingress collision on /a-cong-ocr, /a-cong-ocr-api, or /a-cong-ocr-playground

Check:
scripts/check_k8s_public_ocr.sh

Existing working cluster, UI image replacement only:
chmod +x scripts/*.sh
./scripts/replace_public_ocr_ui_image.sh

Use this replacement script when vLLM and OCR API are already healthy and only web/playground UI changed.
If the loaded UI tar only contains local tag a-cong-ocr-ui:chandra, the script retags it to the Harbor image automatically.
If OCR API code changed too, run: UPDATE_OCR_API_IMAGE=1 ./scripts/replace_public_ocr_ui_image.sh
For a first deploy or rebuild from scratch, use deploy_public_ocr_closed_network.sh instead.

Existing working cluster, first split migration without touching vLLM:
chmod +x scripts/*.sh
./scripts/migrate_public_ocr_split_ui.sh

Use this once when the old cluster already has healthy a-cong-ocr-service and a-cong-vllm-ocr, but not the new lightweight app/playground split yet.
It applies only app/playground deployments, their services, ingress rules, and minimal ConfigMap keys.
It does not restart a-cong-vllm-ocr.

vLLM transformer/qwen rule:
- If scripts/validate_vllm_image_offline.sh fails on the internet/prep PC, the same image will fail in the closed network.
- Do not try pip install inside the closed network.
- Rebuild and re-export the vLLM image on the internet/prep PC with the exact chandra-ocr-2 model folder used in the field.
- The vLLM image contains a startup compatibility alias for qwen2_5 -> qwen2 and qwen2_5_text -> qwen2.
- The included Chandra model is qwen3_5, but it also uses Qwen2VLImageProcessorFast and Qwen2Tokenizer. The preflight script validates all of these before deployment.
- It was tested with Docker --network none, GPU runtime, the included model folder, vLLM serve, and /health.
