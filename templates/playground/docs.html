Model Overview

army-ocr 모델/API

army-ocr은 Chandra OCR 모델을 문서 처리 서버로 감싼 범용 OCR API입니다. PDF와 이미지에서 텍스트뿐 아니라 문서 구조, 블록 좌표, 이미지 영역까지 추출해 검색, RAG, 문서 자동화, 국회 문서 처리 같은 후속 시스템에서 바로 쓸 수 있는 형태로 반환합니다.

Model Chandra OCR 2

문서 이미지 기반 OCR 및 layout block 추출

Inputs PDF / Image

PDF, PNG, JPG, JPEG, WEBP

Outputs JSON / MD / HTML

blocks, bbox, chunks, 이미지 포함 ZIP

이 문서의 주제는 OCR 모델과 범용 OCR API입니다. 국회 OCR은 이 API의 결과를 국회 문서 업무에 맞게 재가공한 특화 레이어입니다. 브라우저 체험 화면은 모델을 테스트하기 위한 보조 UI일 뿐, 외부 연동 계약의 중심은 /api/v1/*입니다.

API 계약

현재 공개된 API Guide는 FastAPI OpenAPI에 실제 등록된 endpoint만 기준으로 작성합니다. Datalab에 있는 전체 제품 API를 그대로 노출하지 않고, 이 OCR 서비스에 구현된 OCR/문서/파일/컬렉션/템플릿/평가/배치/워크플로우 기능만 다룹니다.

Base URL http://14.50.225.33:18110

인터넷망 preview 기준. 운영 ingress에서는 배포 prefix를 base로 사용합니다.

API Prefix /api/v1

외부 연동 계약은 이 prefix 아래의 endpoint를 기준으로 잡습니다.

Auth Preview: none

현재 preview는 별도 key가 없습니다. 운영 공개 시 gateway 인증을 붙이는 구조입니다.

Content Type multipart / json

OCR 업로드는 multipart, 파일 record 기반 후처리는 JSON body를 사용합니다.

Quickstart

가장 빠른 연동은 health 확인 후 /api/v1/marker에 파일을 올리고, 반환된 request_id로 결과를 조회하는 방식입니다.

curl http://127.0.0.1:18110/health
curl http://127.0.0.1:18110/api/v1/capabilities

curl -X POST http://127.0.0.1:18110/api/v1/marker \
  -F "file=@sample.pdf" \
  -F "output_format=json,markdown,html,chunks" \
  -F "page_range=0" \
  -F "mode=balanced"

curl http://127.0.0.1:18110/api/v1/marker/{request_id}

연동 흐름

1. 바로 변환

파일을 한 번 올리고 OCR 결과를 바로 받는 가장 단순한 연동입니다.

POST /api/v1/marker
GET  /api/v1/marker/{request_id}

2. 파일 저장소 기반 처리

파일을 먼저 등록한 뒤 문서 생성, 변환, 분할, 추출에 재사용합니다.

POST /api/v1/files
POST /api/v1/create_document
POST /api/v1/convert_document

3. 구조화 추출

필드 schema를 만들고 OCR 텍스트에서 필요한 값만 뽑습니다.

POST /api/v1/generate_extraction_schemas
POST /api/v1/extract_structured_data

4. 일괄 처리

컬렉션에 여러 파일을 넣고 변환/분할/추출을 batch로 실행합니다.

POST /api/v1/collections
POST /api/v1/batch_runs
GET  /api/v1/batch_runs/{batch_run_id}/results

처리 파이프라인

  1. 입력 수신업로드 파일 또는 file_url을 받습니다.
  2. 페이지 렌더링PDF는 페이지 이미지로 렌더링하고, 이미지는 그대로 사용합니다.
  3. Chandra OCRvLLM 또는 HF runner를 통해 OCR/layout 추론을 실행합니다.
  4. 구조화블록, bbox, 텍스트, Markdown, HTML, chunks를 구성합니다.
  5. 결과 저장요청별 request_id 디렉터리에 결과와 이미지 산출물을 분리 저장합니다.

좌표와 블록

모든 bbox는 렌더링된 페이지 이미지의 픽셀 좌표입니다. 형식은 [x1, y1, x2, y2]이고, page_range는 zero-based, 응답의 page_number는 1-based입니다. 

{
  "label": "text",
  "bbox": [60, 80, 400, 140],
  "text": "문서에서 추출된 본문",
  "confidence": 0.95
}

오류 처리

FastAPI 표준 오류 응답을 사용합니다. 입력 형식 오류, 잘못된 page_range, 존재하지 않는 request_id는 명확한 HTTP 상태 코드와 detail로 반환됩니다.

상태대표 상황대응
400빈 파일, 잘못된 mode/page_range, 필수 입력 누락요청 파라미터와 파일을 수정합니다.
404존재하지 않는 request/file/document/template IDID가 맞는지, 결과가 정리되지 않았는지 확인합니다.
413체험 UI 업로드 크기 제한 초과문서를 분할하거나 운영 API 제한을 조정합니다.
422FastAPI validation 오류필드 타입과 JSON 구조를 확인합니다.
503OCR 서비스 초기화 전health를 재확인하고 vLLM/OCR worker 상태를 점검합니다.

이미지 보존

이미지가 중요한 문서에서는 Markdown에 링크만 남기면 결과 재현성이 떨어집니다. 그래서 OCR 체험 UI의 다운로드는 실제 이미지 파일을 함께 포함한 ZIP으로 제공합니다.

운영과 동시성

단일 vLLM worker 또는 단일 GPU 구성에서는 OCR_MAX_CONCURRENT_REQUESTS=1을 권장합니다. 여러 사용자가 동시에 요청해도 각 요청은 고유 request_id로 분리되고, OCR 모델 호출은 gate에서 순서대로 통과합니다. 병렬 처리량을 늘릴 때는 vLLM --max-num-seqs, GPU 메모리, OCR_MAX_CONCURRENT_REQUESTS를 같이 조정해야 합니다.

timeout, Chandra prompt/batch, playground 기본 쪽수, 업로드 제한, 국회 API 주소, LLM 후처리 주소, 감시 폴더 polling 값처럼 자주 바뀌는 값은 /playground/admin 관리자 페이지 또는 관리자 세션을 가진 /playground/api/admin/runtime-settings에서 저장합니다. Docker/k8s 기본 저장 위치는 /data/runtime/runtime-config/settings.json이며, 이미지를 다시 만들지 않고 새 요청부터 반영됩니다. vLLM 모델 경로, max-num-seqs, context 길이, GPU 메모리, 이미지 pixel limit처럼 restart_required인 값은 저장 후 vLLM Pod만 재시작하면 entrypoint가 저장 JSON을 읽어 반영합니다. 계정 신청과 승인 상태는 /data/runtime/runtime-config/auth.json에 저장됩니다.

체험 UI

/playground/는 모델/API를 직접 호출하기 전에 결과 품질과 출력 형식을 눈으로 확인하는 웹 클라이언트입니다. 제품 설명과 연동 문서는 OCR 모델/API를 기준으로 읽고, UI는 데모와 QA 용도로 사용하면 됩니다. UI는 긴 OCR 요청이 브라우저 요청 하나에 묶이지 않도록 /playground/api/convert/start로 제출하고 /playground/api/convert/{request_id}로 상태를 확인합니다.
